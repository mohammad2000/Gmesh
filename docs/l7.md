# L7 Classifier (Phase 18)

`internal/l7` gives operators per-protocol visibility into mesh
traffic: "what protocols are peers speaking?" and "how does egress
split across HTTP / TLS / SSH / DNS / …?".

## Scope honesty — NOT eBPF

The roadmap sketched Phase 18 as an eBPF L7 classifier. A proper
implementation would load a TC-classifier eBPF program that
inspects TLS ClientHello SNI, HTTP Host headers, QUIC Initial
packets, etc. That's multi-day kernel work: clang + llvm toolchain,
[cilium/ebpf](https://github.com/cilium/ebpf) generators, verifier-
compatible C, kernel-version gating.

What ships today is a **port-based, conntrack-driven classifier in
Go userspace**. The operational value is the same for the common
questions (how much of my mesh is HTTPS? who's running SSH at 3 AM?);
the accuracy is lower (protocols on non-standard ports get
mislabelled). The interface is narrow on purpose — swapping in an
eBPF DPI is a drop-in later without changing the public API.

## How it works

```
           /proc/net/nf_conntrack          (kernel, filled by netfilter)
                    │
                    ▼
            ConntrackReader.Read         (parse + extract 5-tuple + bytes)
                    │
                    ▼
       Classifier.Classify(&flow)        (port → L7 label; mesh-IP → peer_id)
                    │
                    ▼
         Aggregator.Ingest(flows)        (delta accounting per key)
                    │
                    ▼
              Per-(peer, protocol) totals     ───▶  ListL7Totals RPC
              Per-flow classifications        ───▶  ListL7Flows RPC
```

Every 10 s the Monitor:
1. Reads `/proc/net/nf_conntrack` (or a stub for tests / non-Linux).
2. Classifies each flow by destination port (falls back to source
   port, then "unknown" if neither is in the port map).
3. Tags the flow with a `peer_id` if its src or dst IP is a known
   mesh peer.
4. Computes byte deltas vs the previous snapshot and adds them to
   per-(peer, protocol) totals.

### Delta accounting

Conntrack byte counters **can reset** when an entry ages out and a
new one takes its slot on the same 5-tuple. The aggregator guards
against this by only adding the positive delta; a counter that went
backwards contributes 0. Totals therefore monotonically grow and
never subtract real traffic.

### Accounting enablement

The conntrack kernel counters live behind a sysctl:

```bash
sysctl -w net.netfilter.nf_conntrack_acct=1
```

Without it, `/proc/net/nf_conntrack` still lists flows but the
`bytes=` fields are zero. The classifier gracefully reports flows
with zero bytes — totals stay at 0 until `acct` is turned on. The
Linux backend does NOT set this sysctl on behalf of the operator;
it's a host-wide setting and belongs in /etc/sysctl.d.

## Events

Unlike Phase 21 (anomalies), Phase 18 is **poll-only** today. No
event bus signal; operators subscribe to the totals RPC on a cron or
drive a dashboard page refresh. This keeps the event bus uncluttered
by per-flow chatter. A future Phase 18.5 could add `l7_protocol_new`
when the first flow of a given protocol shows up per peer.

## Interaction with other features

- **Anomaly detection (Phase 21)**: the anomaly package uses
  aggregate bytes/sec per peer, not per-protocol. A future anomaly
  detector could subscribe to L7 totals to catch "SSH traffic
  suddenly 10× normal" patterns.
- **Quota (Phase 13)**: quotas are per-egress-profile, not
  per-protocol. If you need "limit HTTPS to 1 GB/month", use an
  egress profile with `dest-ports 443 protocol tcp` and attach a
  quota to that profile.
- **Policy engine (Phase 17)**: no `when: l7_protocol=…` trigger yet.
  Operators poll `ListL7Totals` from an external scheduler if they
  need protocol-aware automation.

## Limitations (honest list)

1. **Port-based classification**. Protocols on non-standard ports
   are mislabelled or show up as "unknown". There is no payload
   inspection, no TLS SNI, no HTTP Host parsing.
2. **No encrypted-within-encrypted visibility**. Traffic already
   encrypted in the mesh WireGuard tunnel is ciphertext to the
   classifier; conntrack sees the OUTER 5-tuple (wg-gmesh port
   52060 ↔ peer's port), not the inner flow. For inner-flow
   visibility the classifier must run on a mesh node that has the
   tunnel terminated — which is where it does run today, on each
   gmesh node.
3. **Conntrack hashtable pressure**. Very high flow rates can fill
   the kernel's conntrack table and start dropping entries. The
   classifier doesn't currently detect this; operators should
   monitor `net.netfilter.nf_conntrack_count` vs `_max`.
4. **IPv4 only**. IPv6 conntrack rows are skipped by the parser.
   When gmesh adds IPv6 support (not yet), the parser will need a
   sibling branch.

## Roadmap

- v0.10.x (this release): port-based classifier shipped.
- v0.11.x: operator-supplied port map via YAML config.
- v1.x: real eBPF TC classifier that reads TLS SNI + HTTP Host,
  preserving the same `Flow` / `PerProtocolTotal` output shape so
  callers don't churn.
